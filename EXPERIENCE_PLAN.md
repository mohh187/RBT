## 0) ما نُفّذ الآن (هذه الجلسة)

| البند | الحالة |
|---|---|
| إصلاح الجوال/التابلت للكاشير | ✅ شبكات CSS مستجيبة لكل قالب، صور `aspect-ratio: 16/10` بدل ارتفاع ثابت، تابلت = عرض ثنائي حتى 640px، هاتف = تكديس مع فاتورة ≤46dvh، الافتراضي على الهاتف «قائمة مدمجة» |
| محرك ثيمات النظام | ✅ `src/lib/systemThemes.js` + سمة `data-systheme` على الأغلفة الأربعة (admin/cashier/kds/pos) |
| 4 ثيمات v1 | ✅ الكلاسيكي (الأصلي) + نوار الليلي (داكن فاخر) + دافئ هادئ (كريمي ناعم) + حاد عملي (مسطّح حاد) |
| مزج لكل قسم | ✅ `tenant.systemTheme` عام + `tenant.systemThemeBy.{admin,cashier,kds}` تجاوز، بنفس آلية سكنات المنيو |

## 1) الثيمات — المرحلة الثانية (لتصبح «أنظمة مختلفة تماماً»)

المرجع: صور POS المرفوعة (CoreChef داكن، Alzera الحاد، الكريمي الدافئ، Yunno البنفسجي، ClaPos الجانبي…).

1. **متغيرات هيكلية لكل ثيم** (JSX وليس CSS فقط): موضع التنقل (شريط جانبي / علوي / سفلي)، فاتورة POS يمين أو أسفل، بطاقات أصناف بصورة كبيرة/صف/دائرية. تنفَّذ كسمات `data-nav`, `data-posbill` يقرأها الغلاف — نفس نمط `data-template` الناجح.
2. **خط لكل ثيم**: نوار = خط عرض أثقل، حاد = أرقام monospace (`--font-body` متغير أصلاً — سهل).
3. **ثيمات إضافية من المراجع**: «بنفسجي حيوي» (Yunno)، «أخضر منعش» (ChainPOS)، «أسود ذهبي» (Pawonku dark).
4. **محرر ثيم داخلي** (مثل محرر سكنات المنيو تماماً): نسخ ثيم → تعديل الألوان/الزوايا/الظلال → حفظ في `tenant.customSystemThemes[]`.
5. **معاينة حية قبل الحفظ**: زر «معاينة» يطبّق السمة مؤقتاً على الصفحة الحالية دون كتابة في Firestore.
6. **تحكم المنصة**: platform DesignTools يطبّق/يقفل ثيم منشأة (نفس آلية قفل سكن المنيو الموجودة).

## 2) جولة الإعداد الأولى (Setup Wizard) + الجولة التعريفية (Tour)

### 2.1 الويزارد بعد التسجيل
الموجود حالياً: تسجيل → إنشاء tenant → دخول مباشر للوحة. الجديد: حقل `tenant.setup = { step, done }` — والـ AdminLayout يعيد التوجيه إلى `/setup` ما دام `!setup.done`.

**الخطوات (5):**
1. **بيانات المنشأة**: الاسم/الشعار/الألوان/العملة/الهاتف (نموذج Settings الحالي مُعاد استخدامه مصغّراً).
2. **الأرقام المهمة**: عدد الطاولات (توليد تلقائي)، أقسام المنيو الأولية، ساعات العمل.
3. **المظهر والهوية**: اختيار ثيم النظام (الرقاقات الجديدة) + سكن المنيو + معاينة حية جنباً إلى جنب — أو «التلقائي والمتابعة».
4. **المراجعة والتفعيل**: ملخص + شروط الاستخدام (نظام /legal الموجود) → `setup.done = true`؛ وإن كانت الموافقة اليدوية مفعلة في المنصة يبقى `pending` وتظهر شاشة انتظار (المنصة توافق من Console — الآلية موجودة في platform admin).
5. **فتح النظام بالمظهر المختار** → إطلاق الجولة تلقائياً.

### 2.2 مكوّن الجولة (Tour Engine) — بدون مكتبات
مكوّن `<Tour steps={...} />` خفيف: طبقة تعتيم + «ثقب» حول العنصر المستهدف (`element.getBoundingClientRect` + مسار SVG أو `box-shadow` عملاق) + بطاقة شرح بموضع ذكي + أزرار (التالي/تخطي/لا تُظهر مجدداً). كل خطوة: `{ id, selector, titleAr, bodyAr, route, media?, action? }`. تقدم المستخدم يُحفظ في `tenant.tours = { [tourId]: { seen, step, at } }` (لكل مستخدم أدق: `users/{uid}.tours`).

**سيناريو الجولة الأساسية (First-run):**
1. الرئيسية: بطاقات اليوم والمبيعات بالساعة.
2. **المساعد الذكي (نجمة الجولة)**: الجولة تكتب مثالاً جاهزاً في حقل الدردشة — «أضف مادة خام: حليب، الوحدة لتر، المخزون 40، حد التنبيه 10» — المستخدم يضغط إرسال والمساعد ينفّذ فعلياً (actions.js موجود وينفذ)، ثم الجولة تنتقل إلى **المخزون** وتُظهر المادة المضافة + شرح الطريقة اليدوية بجانبها.
3. عودة للمساعد بمثال ثانٍ: «أعطني تقرير مبيعات مختصر لهذا الأسبوع» → عرض الرد → إشارة لأمثلة أخرى (سؤال عن موظف، اقتراح استراتيجية عروض).
4. المنيو: إضافة صنف تجريبي (يُحذف آخر الجولة أو يُعلَّم «تجريبي»).
5. الكاشير: فتح POS، إضافة صنف، **بيع سريع** حقيقي على طلب تجريبي.
6. رحلة الطلب: الانتقال للمطبخ ورؤية التذكرة → جاهزة → قُدّم (يشرح KPI والطابور).
7. الإعدادات والمظهر: أين يغيّر الثيم والقوالب لاحقاً.
8. الختام: بطاقة «ماذا بعد؟» + رابط إعادة الجولة من الإعدادات.

**تنويه أمان:** أمثلة المساعد في الجولة تُنفّذ على بيانات المنشأة الحقيقية — تُعلَّم `demo: true` وتظهر أداة «تنظيف بيانات الجولة» في نهايتها.

## 3) إدارة التيتوريال من المنصة المركزية (/platform/tutorials)

### 3.1 نموذج البيانات
```
platformTutorials/{id}: {
  titleAr/En, steps[] (نفس صيغة Tour أعلاه: selector/route/media/action),
  trigger: 'first_login' | 'feature_launch' | 'manual' | 'route_visit',
  route?: '/admin/inventory',              // للـ route_visit
  frequency: 'once' | 'until_dismissed' | 'every_time' | 'once_per_version',
  version: 3,                              // رفع الرقم يعيد إظهاره لمن شاهد نسخة أقدم
  audience: { mode: 'all' | 'plans' | 'venues', plans?: [], venueIds?: [] },
  featureKey?: 'inventory',                // يظهر فقط لمن خطته تشمل الميزة
  active, publishAt?, expireAt?, priority,
  stats: { impressions, completions, dismissals }   // تُجمّع من الأحداث
}
```
العميل: عند تحميل AdminLayout يجلب التيتوريالات الفعّالة المطابقة (خطة/منشأة/مسار) ويقارن مع `users/{uid}.tours` → يعرض الأعلى أولوية. **قراءة فقط من جهة المنشأة** (rules: read للجميع الموثقين، write للمنصة فقط).

### 3.2 المحرر في المنصة
- بناء الخطوات ببطاقات (عنوان/نص/صورة أو GIF أو فيديو قصير/المسار/محدد العنصر) مع **مكتبة محددات جاهزة** (نسمّي العناصر المهمة بـ `data-tour="pos-open"` في الكود مرة واحدة — أثبت من CSS selectors).
- معاينة حية: زر «جرّب كمنشأة» يفتح الجولة على حساب تجريبي (آلية الانتحال impersonation موجودة أصلاً).
- إطلاق ميزة جديدة = قالب جاهز «إعلان + جولة»: بانر داخل اللوحة (نظام البث broadcasts الموجود) يقود إلى الجولة.
- الاستهداف: الكل / خطط محددة / منشآت محددة (بحث متعدد) / نسبة تدريجية % (staged rollout).
- تحليلات: نسبة الإكمال لكل خطوة → تحديد الخطوة التي يتسرب عندها المستخدمون.

### 3.3 توسعات ذكية مقترحة
تلميحات سياقية صغيرة (tooltip لمرة واحدة) بلا جولة كاملة؛ «مركز التعلم» داخل اللوحة يجمع كل الجولات لإعادة تشغيلها؛ ربط بالمساعد الذكي («اشرح لي هذا القسم» يشغّل الجولة)؛ تنبيه المنصة عند انخفاض إكمال جولة عن حد؛ جولات بحسب الدور (كاشير يرى جولة الكاشير فقط).

## 4) شاشات العرض (Digital Signage) — القسم الجديد

### 4.1 البنية (آلية الربط)
- **الشاشة = صفحة ويب**: مسار عام `/screen/{screenId}` يفتح على أي جهاز بمتصفح (تلفزيون ذكي/Chromecast/تابلت/كمبيوتر موصول بشاشة). لا حاجة لأجهزة خاصة أو واي فاي مشترك — **Firestore realtime هو قناة الربط**، فأي تعديل من اللوحة يظهر على الشاشة لحظياً.
- **الاقتران مثل تلفزيونات التطبيقات**: الشاشة تعرض رمز اقتران (6 أحرف) → المدير يدخله في «الشاشات» بالإدارة → توثيق الشاشة وربطها (`tenants/{tid}/screens/{id}`: name, pairedAt, playlist, layout, lastSeen heartbeat).
- **المحتوى**: `tenants/{tid}/signage/playlists/{id}` — عناصر مرتبة: {type: menu|image|video|offer|orders-board|custom-html, duration, transition}. الوسائط عبر Storage الموجود (uploadImage/uploadFile).
- **طبقة الطلبات الحية**: خيار «إشعارات الطلبات فوق المحتوى» — الشاشة تراقب `watchActiveOrders` وتعرض توست «طلبك جاهز 0798» فوق الفيديو/الخلفية (نمط شاشات الاستلام في المطاعم).
- **ربط المنيو والعروض تلقائياً**: عنصر «منيو» يعرض الأصناف من Firestore مباشرة — أي تعديل سعر ينعكس فوراً؛ وعند إنشاء عرض/كود خصم في قسم العروض يظهر خيار **«انشر على شاشات العرض»** يضيف بطاقة العرض للبلاي ليست بضغطة.
- **معاينة حية في اللوحة**: iframe لنفس مسار الشاشة داخل محرر البلاي ليست.

## 5) ثيمات الطاولات (Table View)

الموجود: قسم الطاولات قائمة/شبكة بسيطة. المرجع: مخطط أرضية بحالات ملونة (متاح/محجوز/مفوتر/مشغول).

1. **نموذج بيانات**: `tables/{id}` يُضاف له `{ x, y, shape: 'round'|'square'|'rect', seats, zone }` — كلها اختيارية فلا يتأثر أي استخدام حالي للطاولات (الكاشير يقرأ label فقط).
2. **قالب «مخطط الأرضية»** يُضاف كخيار جديد في systemTemplates قسم tables: `{list | grid | floor}` — نفس البنية المجربة.
3. **محرر السحب**: وضع «تحرير» يسحب الطاولات على شبكة نقاط (dnd-kit موجود ومستخدم) ويحفظ الإحداثيات، مناطق (صالة/خارجية) كتبويبات.
4. **الحالة الحية بالألوان**: متاح (أخضر)، مشغول (رمادي — عليه طلب نشط)، مفوتر (أزرق — طلب served غير مدفوع)، محجوز (أصفر — من نظام الحجوزات الموجود). تُشتق من الطلبات الحية بلا حقول جديدة.
5. **إجراء باللمس**: لمس طاولة متاحة → فتح POS عليها (الموجود initialTable يدعمها)؛ مشغولة → تفاصيل الطلب؛ مفوترة → شاشة الدفع.
6. **ثيمات المخطط**: يرث ثيم النظام تلقائياً (نوار/دافئ/حاد) — لا عمل إضافي بفضل المتغيرات.
7. مرحلة لاحقة: دمج طاولات، حجز من المخطط، خريطة حرارية للاستخدام.

## 6) قفل النظام بـ PIN خفيف (لكل موظف)

**الفكرة**: جلسة الجهاز تبقى مسجلة بحساب المنشأة، لكن فتح الشاشة يتطلب اختيار الموظف + 4 أرقام — سرعة تبديل على نفس الجهاز + مساءلة (من نفّذ ماذا).

1. **البيانات**: `tenants/{tid}/staff/{id}` يُضاف `pinHash` (SHA-256 للـ PIN + ملح tenantId عبر WebCrypto — لا يُخزن نصاً). تعيين/تغيير الـ PIN من إدارة الفريق (صلاحية المدير) أو الموظف نفسه بعد توثيق.
2. **شاشة القفل** `<PinLock/>`: تظهر فوق النظام عند: تفعيل الخيار + (بدء الجلسة / خمول قابل للضبط 1-30د / زر قفل يدوي في الشريط). تعرض الموظفين النشطين كبطاقات بأسمائهم → لوحة أرقام كبيرة → تحقق محلي من الهاش.
3. **الأثر**: عند الفتح يُسجَّل `activeStaff` في localStorage ويُستخدم اسمه تلقائياً كـ `actorName` في الكاشير/الطلبات (الحقل موجود ومستخدم فعلاً في createOrder/payOrder) — فتصبح المساءلة حقيقية بلا أي تغيير في منطق الطلبات.
4. **إعدادات**: تفعيل/تعطيل، مدة الخمول، الشاشات المشمولة (كاشير فقط أو كل اللوحة)، سماح «مدير فقط» لأقسام حساسة (الإعدادات/التقارير) بطلب PIN مدير.
5. **حدود أمنية (صريحة)**: هذا **قفل تشغيلي ضد العبث على جهاز مفتوح**، ليس تشفيراً — من يملك بيانات دخول Firebase يتجاوزه. الأمان الحقيقي يبقى بالحساب والقواعد. 5 محاولات خاطئة = تأخير تصاعدي + تنبيه للمدير.
6. **قواعد**: كتابة pinHash لصاحب صلاحية إدارة الفريق فقط؛ لا يُقرأ في واجهات الزبائن.

## 7) ضمانات عدم الكسر (تسري على كل ما سبق)

- كل ميزة جديدة: **حقول اختيارية** على وثائق موجودة أو **مجموعات جديدة** — لا تعديل بنية قائمة.
- كل واجهة جديدة: مسار/بطاقة جديدة خلف planAllows أو flag — الشاشات الحالية لا تتغير سلوكياً.
- الثيمات: CSS فقط عبر سمة قابلة للإزالة؛ القيمة `classic` = بلا سمة = المظهر الحالي بالبت.
- بعد كل خطوة: `npm run build` + فحص المسارات الأساسية (كاشير/مطبخ/طلبات/إعدادات).
- أولوية التنفيذ المقترحة: (1) صقل الثيمات هيكلياً + ثيمان إضافيان → (2) ويزارد الإعداد + الجولة الأساسية → (3) قفل PIN (صغير وقيمة فورية) → (4) مخطط الطاولات → (5) شاشات العرض M1 → (6) إدارة التيتوريال المركزية (تعتمد على محرك الجولة من 2).
