- لا يوجد **تنوّع قوالب** للباك-أوفيس: كل المنشآت ترى نفس تخطيط الكاشير/المطبخ/الداشبورد. هذا هو جوهر طلبك.

## 2. الرؤية: «قوالب لكل قسم» (Back-office Templates)

مثلما للمنيو سكنات، نبني **نظام قوالب** لكل سطح رئيسي في النظام — تختار المنشأة القالب الذي يناسب طاقمها ومساحتها وأجهزتها. القوالب **تابعة للباقة** (المجانية: قالب واحد افتراضي؛ المتقدمة: كل القوالب + تخصيص).

### القوالب المقترحة لكل قسم

| القسم | القوالب المقترحة | لماذا |
|---|---|---|
| **الكاشير (POS)** 🔴 | **1) شبكة أصناف كلاسيكية** (تايلات كبيرة لمس) · **2) قائمة مدمجة سريعة** (بحث + قائمة كثيفة للكيبورد) · **3) لمس-متجر** (فئات جانبية + شبكة صور) · **4) خفيف (سطر واحد)** لأجهزة صغيرة | كل مقهى مختلف: شاشة لمس كبيرة مقابل جهاز صغير مقابل كاشير سريع بالكيبورد |
| **المطبخ (KDS)** 🔴 | **1) سكة تذاكر** (Ticket rail أفقي) · **2) أعمدة كانبان** (جديد/تحضير/جاهز) · **3) شبكة بطاقات** كبيرة · **4) شاشة عرض** (Bump bar، خط إنتاج) | المطبخ يحتاج وضوحاً لحظياً؛ الفريق يفضّل تدفقات مختلفة |
| **الداشبورد** | **1) تنفيذي** (KPI كبيرة + رسوم) · **2) تشغيلي** (طلبات حيّة + مهام) · **3) مبسّط** | المالك يريد أرقاماً؛ المناوب يريد التشغيل |
| **المنيو (إدارة)** | **1) جدول كثيف** · **2) بطاقات بصور** · **3) كتالوج** | كتالوج كبير مقابل منيو صغير مصوّر |
| **الطلبات** | **1) لوحة كانبان** · **2) قائمة زمنية** · **3) خريطة طاولات** | |
| **العمليات/الطاولات** | **1) خريطة أرضية** · **2) شبكة طاولات** · **3) قائمة** | |
| **العملاء/الطاقم/التقارير** | قالب «تحليلي» + قالب «قائمة» لكل منها | |

### معمارية نظام القوالب (كيف ننفّذه تقنياً)
1. حقل على المنشأة: `systemTemplates: { cashier:'grid'|'compact'|'touch'|'lite', kds:'rail'|'kanban'|'grid'|'display', dashboard:'exec'|'ops'|'min', ... }`.
2. `AdminLayout`/الشاشات تقرأ القالب وتضع `data-template` على الحاوية (كما تفعل السكنات مع المنيو).
3. لكل قالب: كتلة CSS في نظام التصميم + (عند اللزوم) فرع تخطيط في المكوّن (مثل `CashierPOS` يبدّل بين grid/list حسب `data-template`).
4. **بوابة الباقة:** `planAllows(tenant, 'systemTemplates')` (Pro+)؛ المجانية على الافتراضي.
5. شاشة اختيار في الإعدادات (معاينة حية لكل قالب — نعيد استخدام نمط معاينة الثيمات).
6. تحكّم المنصة: يمكن لفريقنا تطبيق/قفل قالب لأي منشأة (نعيد استخدام `DesignTools`/`platformUpdateTenant`).

## 3. خطة إعادة التصميم قسماً بقسم (باستخدام المهارات)

لكل قسم نمرّ بالحلقة: **مرجع بصري → توجيه → بناء (impeccable) → صقل (emil) → مراجعة**.

### أولوية 1 — الكاشير والمطبخ (الأهم لطلبك)
- **الكاشير `CashierPOS.jsx`:** إعادة بناء كـ 4 قوالب قابلة للاختيار؛ تحسين: أزرار لمس أكبر (≥44px)، أرقام tabular، حركة إضافة صنف للسلة، إيصال جانبي واضح، اختصارات كيبورد للقالب المدمج، حالات فارغة/خطأ.
- **المطبخ `Kds.jsx`:** قوالب rail/kanban/grid/display؛ ألوان حالة قوية، عدّاد زمن لكل تذكرة (تحذير عند التأخير)، حركة «bump» عند الإنجاز، وضع ملء الشاشة، تباين عالٍ لبيئة المطبخ.

### أولوية 2 — الداشبورد والمنيو والطلبات
- **الداشبورد `Dashboard.jsx`/`InsightsHub`:** 3 قوالب؛ KPI بارزة، رسوم أنظف (recharts موجود)، رتم مسافات، حالة «لا بيانات» أنيقة.
- **إدارة المنيو `Items/Categories/MenuHub`:** قوالب جدول/بطاقات/كتالوج، رفع صور أسرع، سحب وإفلات (dnd-kit موجود).
- **الطلبات `Orders.jsx`:** كانبان/زمني/طاولات.

### أولوية 3 — البقية
العمليات/الطاولات، العملاء، الطاقم/HR، التقارير، الإعدادات، الشكاوى، المخزون — صقل بصري موحّد + قالب تحليلي/قائمة حيث يفيد.

### الأساس المشترك أولاً (قبل الأقسام)
1. **تحديث نظام التصميم (`index.css`)** بمعايير المهارات: سلّم ألوان OKLCH، خطوط بشخصية (Geist/Outfit للاتيني + الحفاظ على الخط العربي)، ظلال ملوّنة، سلّم z-index دلالي، حركة ease-out أسّية + `prefers-reduced-motion`، أرقام tabular للبيانات.
2. **إنشاء `PRODUCT.md` + `DESIGN.md`** (يطلبهما skill impeccable) لتثبيت الهوية والقرارات.
3. **مكوّنات أساس موحّدة** (أزرار، حقول، بطاقات، حالات فارغة/تحميل/خطأ، Sheet) مصقولة مرة واحدة تنعكس على كل الشاشات.

## 4. المراحل الزمنية
- **م0 — الأساس:** PRODUCT/DESIGN.md + ترقية `index.css` + مكوّنات الأساس + بنية نظام القوالب (data-template + بوابة الباقة + شاشة الاختيار). 
- **م1 — الكاشير + المطبخ** (قوالبهما الكاملة). ← أولوية طلبك
- **م2 — الداشبورد + المنيو + الطلبات.**
- **م3 — بقية الأقسام + منيو الزبون (تحسين إضافي فوق السكنات).**
- **م4 — صقل شامل (emil/review-animations) + فحص وصولية/أداء + معاينة على أجهزة حقيقية.**

**بعد كل شاشة:** بناء (`npm run build`) + تشغيل/لقطة للتأكد بصرياً قبل الانتقال.

## 5. ملاحظات تنفيذ مهمة
