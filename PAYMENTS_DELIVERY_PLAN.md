# خطة ماكينة دفع الشبكة (مدى) + التوصيل الشامل — دراسة معمقة

> الهدف الأول: عند اختيار «دفع بالشبكة» في الكاشير يظهر المبلغ **تلقائياً** على جهاز الدفع، وعند إتمام العملية يقرأ النظام النجاح **تلقائياً** ويقفل الطلب مدفوعاً — بلا إدخال يدوي وبلا أخطاء مطابقة.
> الهدف الثاني: التوصيل كنوع طلب كامل في المنيو والكاشير والطلبات والتقارير، مع سائقين وتتبع ورسوم مناطق.

---

## الجزء الأول: ربط ماكينة الدفع (ECR Integration)

### 1) كيف تُربط ماكينات الدفع فعلياً — أربعة نماذج

| النموذج | كيف يعمل | المتطلبات | ملاءمته لنا |
|---|---|---|---|
| **A. سحابي (Cloud Terminal API)** | نرسل «نية دفع» عبر REST API للمزود → سحابة المزود توقظ الجهاز وتعرض المبلغ → إشعار Webhook عند الدفع | جهاز أندرويد ذكي من المزود (Sunmi/PAX) متصل بالإنترنت | **الأفضل لنا** — متصفح فقط، لا برامج محلية، يعمل مع Firebase Functions مباشرة |
| **B. محلي (LAN/ECR bridge)** | الكاشير يرسل المبلغ للجهاز عبر الشبكة المحلية (بروتوكول ECR للبنك/المزود عبر TCP) | وسيط محلي لأن المتصفح لا يفتح TCP: خدمة صغيرة على جهاز الكاشير أو جهاز طرفية يدعم HTTP محلي | ممكن كخيار ثانٍ للمنشآت ذات إنترنت ضعيف؛ يتطلب تثبيت وسيط |
| **C. SoftPOS (App-to-App)** | جهاز الكاشير الأندرويد نفسه هو الطرفية (NFC): نفتح تطبيق المزود بـ Deep Link ومعه المبلغ والمرجع → يُتم العميل الدفع باللمس → يعود التطبيق لنظامنا بالنتيجة + Webhook تأكيدي | جوال/تابلت أندرويد بـ NFC + تطبيق المزود + حساب تاجر | **أسرع طريق للانطلاق** — صفر أجهزة إضافية، مثالي لتابلت الكاشير |
| **D. شبه-تلقائي (Manual-confirm fallback)** | النظام يعرض المبلغ للكاشير، الكاشير يدخله في أي ماكينة بنك عادية، ثم يؤكد النجاح بضغطة (مع سبب إلزامي عند الإلغاء وسجل تدقيق) | لا شيء | موجود عملياً اليوم؛ يبقى كخطة طوارئ عند انقطاع أي ربط |

### 2) مصفوفة المزودين في السعودية (تُتحقق تفاصيلها عند فتح حساب التاجر)

| المزود | النوع | نقاط القوة | طريقة الربط المتوقعة |
|---|---|---|---|
| **Nearpay** (سعودية) | SoftPOS + طرفيات + بوابة ربط | متخصصة في «الربط كخدمة»، SDK أندرويد/iOS وواجهات ويب، معتمدة مدى، وثائق مطورين جيدة | C (SoftPOS SDK/App) + A (Terminal API) + Webhooks |
| **Geidea** (سعودية) | مُحصِّل مرخص + طرفيات ذكية + SoftPOS | الأوسع انتشاراً للمنشآت الصغيرة، جهاز + حساب تاجر من جهة واحدة، لديها ECR للربط مع أنظمة الكاشير | A على أجهزتها الذكية + B (ECR) + C |
| **Neoleap (الراجحي)** | ذراع مدفوعات الراجحي | أكثر الماكينات انتشاراً في المقاهي فعلياً؛ إن كانت ماكينة العميل الحالية منها فالربط يبدأ من البنك | B (mada ECR standard عبر البنك) + A على الأجهزة الذكية |
| **MyFatoorah** | PSP خليجي + ماكينات | API موحد أونلاين + POS، تسوية موحدة | A + Webhooks |
| **معيار mada ECR الوطني** | مواصفة موحدة من المدفوعات السعودية | أي ماكينة بنكية حديثة يمكن للبنك تفعيل «وضع الربط» فيها | B (يتطلب وسيطاً محلياً) |
| **Moyasar** (مستخدم حالياً) | PSP أونلاين فقط | **يبقى كما هو** للدفع الإلكتروني في المنيو (مفاتيح LIVE مفعلة) — لا يوفر طرفيات مادية | لا ينطبق على الماكينة؛ يُكمل المنظومة أونلاين |

**التوصية:** طبقة تجريد داخلية واحدة + مزودان في البداية: **Nearpay SoftPOS** (انطلاق فوري بلا أجهزة) ثم **Geidea/Neoleap Cloud Terminal** لمن يملك ماكينة. القرار النهائي بعد التحقق التجاري (رسوم، متطلبات السجل، مدة التفعيل).

### 3) المعمارية على منظومتنا (Firebase) — مستقلة عن المزود

```
tenants/{tid}/terminals/{termId}
  { provider: 'nearpay'|'geidea'|..., label: 'كاشير 1', deviceRef, active, createdAt }

tenants/{tid}/payIntents/{intentId}
  { orderId, amount, currency, terminalId, method: 'card',
    status: 'pending'|'processing'|'paid'|'failed'|'cancelled'|'timeout',
    providerRef, approvalCode?, cardScheme?, last4?, error?, byName, createdAt, updatedAt }
```

**التدفق:**
1. الكاشير في PaymentSheet يختار «شبكة» → النظام ينشئ `payIntent` (pending) — **المبلغ يُرسل تلقائياً، لا إدخال يدوي**.
2. Cloud Function (onCreate) تستدعي API المزود لتوقظ الماكينة بالمبلغ والمرجع → status = processing.
3. شاشة الكاشير تراقب الوثيقة لحظياً (onSnapshot): «بانتظار الدفع على الجهاز…» + مؤقت + زر إلغاء.
4. Webhook المزود (HTTPS Function) يتحقق من التوقيع → intent = paid + approvalCode/last4 → **معاملة payOrder الموجودة** تُنفذ بنفس منطقها الحالي (method: 'card', ref) → توست نجاح + طباعة تلقائية للفاتورة.
5. فشل/مهلة → الحالة تظهر فوراً مع إعادة المحاولة أو التحويل للنموذج D.

**لماذا هذا التصميم قوي:** لا لمس لبيانات البطاقة إطلاقاً (الشهادات الأمنية كلها على المزود)، إضافة مزود جديد = Function adapter جديدة فقط، وواجهة الكاشير لا تتغير مهما تغير المزود.

### 4) اعتبارات حرجة
- **التسوية والمطابقة:** تقرير يومي يقارن مجموع intents المدفوعة مع تقرير تسوية المزود — أي فرق يظهر كتنبيه.
- **الاسترداد:** يبدأ من شاشة الطلب (refund الموجود) → Function تستدعي refund المزود → لا استرداد يدوي منفصل عن السجل.
- **تعدد الماكينات:** ربط ماكينة لكل كاشير/فرع، مع اختيار افتراضي لكل جهاز (localStorage).
- **الأمان:** مفاتيح المزود في functions/.env فقط (كالمعتاد — لا تُرفع أبداً)، تحقق توقيع Webhook إلزامي، صلاحية إدارة الماكينات للمدير فقط.
- **وضع الطوارئ:** انقطاع إنترنت الماكينة → تحويل بضغطة للنموذج D مع وسم «تأكيد يدوي» في السجل.

## الجزء الثاني: التوصيل — في المنيو والنظام بالكامل

### 1) نموذج البيانات (حقول اختيارية — لا كسر لأي شيء قائم)
```
order.orderType += 'delivery'
order.delivery = { address: { text, lat?, lng?, shortCode? },  // العنوان الوطني المختصر مدعوم
                   phone, notes?, fee, zoneId?, driverId?, driverName?,
                   status: 'pending'|'assigned'|'out_for_delivery'|'delivered'|'failed',
                   assignedAt?, pickedAt?, deliveredAt?, proof? }
tenants/{tid}/deliveryZones/{id} = { name, fee, minOrder, etaMinutes, active, order }
tenants/{tid}/staff — role 'driver' جديد + شاشة «توصيلاتي»
tenant.deliverySettings = { enabled, hours?, defaultFee, freeAbove?, cod: true, online: true }
```

### 2) رحلة الزبون في المنيو
اختيار «توصيل» بجانب (محلي/سفري/استلام) → إدخال العنوان (نصي + زر «موقعي الحالي» geolocation + حفظ عناوين الزبون برقم جواله) → اختيار/تحديد المنطقة تلقائياً → رسوم التوصيل تُضاف للسلة بوضوح مع الحد الأدنى → الدفع: **إلكتروني عبر Moyasar المفعّل أصلاً أو كاش عند الاستلام (COD)** → صفحة تتبع الطلب الحالية تكسب حالة «في الطريق إليك» + اسم/جوال السائق.

### 3) رحلة التشغيل
الطلب يظهر في الكاشير/المطبخ كالمعتاد بوسم توصيل بارز + العنوان → عند «جاهز» يظهر في **لوحة التوصيل**: تعيين سائق (أو التقاط ذاتي من بوابة السائق) → السائق من جواله (PWA + PIN الموجود): «استلمت» → out_for_delivery → «سلّمت» (+ إثبات: صورة أو OTP من الزبون) → إن كان COD تدخل القيمة في **عهدة السائق** وتُسوى في تقرير نهاية اليوم (درج الكاشير الموجود يكسب قسم عهد السائقين).

### 4) مصفوفة تكاملات التوصيل الخارجية

| الجهة | النوع | الواقع العملي |
|---|---|---|
| **سائقون ذاتيون** | داخلي | **المرحلة الأولى** — كل الأدوات عندنا (أدوار، PIN، إشعارات، تقارير) |
| **Barq / iMile / Aymakan** | شركات توصيل بـ API | ربط «اطلب سائقاً» للطلب الواحد — الأنسب بعد المرحلة الأولى |
| **OTO / Torod** | مجمّعات شحن سعودية | API واحد يفتح عدة شركات توصيل — يختصر التكاملات |
| **Mrsool for Business** | توصيل عند الطلب | مناسب للمشاوير الفردية؛ يتطلب حساب أعمال |
| **Jahez / HungerStation / Keeta / ToYou** | منصات سوق (Marketplace) | واجهاتها **بشراكة تجارية فقط** — تُدرس لاحقاً كقناة طلبات واردة، وليست توصيلاً لطلباتنا |
| **واتساب** (مخطط قائم) | إشعارات | رابط التتبع + تأكيد التسليم للزبون — يركب على خطة HOSTING_EMAIL_WHATSAPP |

## العشرون اقتراحاً

**الدفع (1–10):**
1. **طبقة تجريد المزودين** — adapter لكل مزود خلف واجهة واحدة (createIntent/cancel/refund/verifyWebhook)؛ تبديل المزود دون لمس واجهات.
2. **شاشة انتظار حية في الكاشير** — عدّاد، حالة الجهاز، إلغاء، وتحويل فوري لوضع الطوارئ اليدوي الموثق.
3. **مطابقة يومية آلية** — تقرير Z يقارن Firestore مع تسوية المزود ويعلّم الفروقات.
4. **تقسيم الفاتورة على وسائل** — جزء شبكة + جزء كاش في نفس الطلب (payPartial الموجود يمهد لذلك).
5. **إكرامية على الماكينة** — شاشة الجهاز تعرض خيار إكرامية تُسجل في حقل tip الموجود.
6. **ربط ماكينة لكل جهاز كاشير** — واختيار تلقائي بالذاكرة المحلية؛ لا التباس في الفروع.
7. **استرداد من نفس الشاشة** — refund المزود يُستدعى من نافذة الطلب مباشرة مع صلاحية مدير.
8. **مؤشر صحة الماكينة** — آخر اتصال/عملية لكل طرفية في الإعدادات + تنبيه انقطاع.
9. **QR-على-الماكينة كوسيلة إضافية** — نفس intent يولد رمز دفع (مدى أونلاين/STC Pay عبر Moyasar) يظهر على شاشة الطرفية أو شاشة العرض.
10. **سجل تدقيق مالي موحّد** — كل intent بحالاته وطوابعه وأسماء المنفذين — جاهز للمحاسبة وضريبة القيمة المضافة.

**التوصيل (11–20):**
11. **محرر مناطق بالرسوم** — دوائر نصف قطر حول الفرع (بسيطة وفعالة) مع رسوم/حد أدنى/زمن لكل حلقة؛ لاحقاً مضلعات على خريطة.
12. **العنوان الوطني المختصر** — حقل Short Address (مثل RRRD2929) مع تحقق من صيغته؛ أدق من الوصف النصي في السعودية.
13. **بوابة السائق بلا تطبيق** — PWA بمسار /driver + قفل PIN الموجود: توصيلاتي، اتجاهات (رابط خرائط)، اتصال بالزبون بضغطة.
14. **إثبات تسليم** — صورة أو رمز OTP يقوله الزبون؛ يقفل النزاعات.
15. **عهدة COD للسائق** — عداد كاش لكل سائق يسوّى في إغلاق اليوم؛ يركب على درج الكاشير الحالي.
16. **تتبع حي للزبون** — السائق يشارك موقعه (Geolocation → Firestore) وصفحة التتبع ترسم تقدمه؛ بلا أي خدمة خرائط مدفوعة.
17. **إسناد ذكي** — اقتراح السائق الأقل حمولة/الأقرب، مع «التقاط ذاتي» كخيار افتراضي أبسط.
18. **تنبيهات SLA للتوصيل** — طلب في الطريق أطول من X دقيقة → تنبيه للمدير (alertParty الموجود).
19. **توصيل مجدول** — «وصّلوه الساعة 6» — يركب على حقول الحجز/الوقت الموجودة في الطلب.
20. **وضع الازدحام** — زر واحد يرفع رسوم/أزمنة التوصيل مؤقتاً أو يغلق التوصيل مع رسالة مهذبة في المنيو.

## ترتيب التنفيذ المقترح
1. **توصيل MVP** (لا يعتمد على أي جهة خارجية): نوع الطلب + العنوان + المناطق والرسوم + دور السائق + بوابته + COD + التتبع الأساسي. **يمكن البدء فوراً.**
2. **الدفع نموذج D المحسّن** (فوري): زر «شبكة — تأكيد يدوي» الموثق بسجل كامل، ريثما يجهز الربط.
3. **SoftPOS (Nearpay)**: يتطلب منك فتح حساب تاجر لدى المزود — بعدها الربط أيام قليلة على المعمارية أعلاه.
4. **Cloud Terminal (Geidea/Neoleap)** لمن يملك ماكينة + المطابقة اليومية.
5. التكاملات الخارجية للتوصيل (Barq/OTO) ثم دراسة منصات السوق.

## ما أحتاجه منك قبل تنفيذ الربط الفعلي
- أي ماكينة تملك حالياً في المقهى (البنك/المزود والموديل)؟ — يحدد المسار A أو B أو C.
- فتح حساب تاجر لدى المزود المختار (Nearpay أو Geidea) — الربط التقني جاهز التصميم أعلاه.
- تأكيد أن التوصيل بسائقين ذاتيين أولاً (التوصية) أم تريد شركة توصيل من اليوم الأول.
