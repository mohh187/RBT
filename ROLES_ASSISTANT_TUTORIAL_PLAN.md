## القسم الأول — دراسة الأدوار والصلاحيات (المحور الرئيسي)

### 1.1 الوضع الحالي (كما هو في الكود)

**طبقة الواجهة (`permissions.js`):** يوجد 20 صلاحية (CAP) دقيقة:
`take_orders, cancel_order, refund, print, kitchen, scan_tickets, manage_menu, manage_offers,
manage_events, manage_tables, view_reports, view_customers, view_complaints, manage_staff,
manage_payroll, view_performance, manage_settings, attendance, deliver`.

- الأدوار الجاهزة: `owner, manager` (كل الصلاحيات)، ثم `supervisor, cashier, barista, waiter, kitchen, driver, cleaner, staff`.
- `can(cap)` في auth.jsx = `roleCan(profile.role, profile.caps, cap, tenant.roleCaps)`.
- **صفحة الأدوار (`Roles.jsx`)** تكتب `tenant.roleCaps` — أي تخصيص على مستوى **الدور** (كل الكاشيرات معاً). تعمل فعلياً.
- **مسارات `App.jsx`** محمية بـ `<RequireCap cap=...>` تعيد التوجيه إلى `/portal` عند نقص الصلاحية.

### 1.2 الأعطال المؤكَّدة (هذه هي المشكلة الجوهرية)

**العطل A — التخصيص الفردي لكل موظف غير موصول إطلاقاً (dead code):**
- `capsFor()` يدعم `overrides.add/remove` لكل موظف، لكن مصدره `users/{uid}.caps`.
- قواعد Firestore (سطر 65-76) تمنع الموظف من كتابة `caps` على وثيقته، **وتمنع المدير من كتابة وثيقة `users` لموظف آخر** (فقط `platformAdmin`). النتيجة: **لا يوجد أي مسار يكتب `users/{uid}.caps`** عدا سوبر-أدمن المنصّة.
- `StaffProfile.jsx` (محرّر الموظف) فيه تبويبات: نظرة عامة/أداء/رواتب/دوام/إجازات — **ولا يوجد أي محرّر صلاحيات**.
- الخلاصة: **لا يمكن للمدير حالياً إعطاء موظف بعينه صلاحية تختلف عن دوره.**

**العطل B — قواعد Firestore تعرف مستويين فقط (staff / manager)، فالصلاحيات الدقيقة «مسرحية واجهة»:**
- كل الكتابة الحسّاسة تُحرَس بـ `isManager(tid)` فقط: `items, categories, offers, tables, events,
  campaigns, expenses, suppliers`، وإعدادات المنشأة (`tenants/{tid}`).
- لو أعطيت `supervisor` صلاحية `manage_menu` من صفحة الأدوار، **ستظهر له واجهة تحرير المنيو لكن Firestore سيرفض الحفظ** لأنه ليس owner/manager. أي الصلاحية تُظهر الزر ولا تُنفّذ.

**العطل C — مشكلة موظف التسويق (طلبك المباشر):**
- لتمكين موظف من الحملات/العملاء/الولاء/الاستوري، القاعدة الوحيدة التي تسمح بكتابة `campaigns`
  و`tenant` هي `isManager`. أي يجب ترقيته إلى **مدير** → وعندها يرى **كل شيء** (الرواتب، الإعدادات، الأرباح).
- **لا يوجد حل وسط اليوم.** هذا بالضبط ما تريد إصلاحه.

**العطل D — الصفحة الافتتاحية بعد الدخول لا تراعي الدور:**
- بعد الدخول، اللوحة الافتراضية `/admin` تعرض Dashboard بأرقام الإيراد لأي موظف يملك الوصول، دون حارس `VIEW_REVENUE` منفصل.

### 1.3 التصميم المقترح (كامل، قابل للتنفيذ)

#### أ) توسيع سجل الصلاحيات — تقسيم الصلاحيات الضخمة

| صلاحية جديدة | تفصل عن | تعطي |
|---|---|---|
| `MANAGE_CAMPAIGNS` | manage_settings | الحملات والإعلانات فقط (موظف التسويق) |
| `MANAGE_LOYALTY` | manage_settings | العضويات/الولاء/بطاقات VIP |
| `MANAGE_APPEARANCE` | manage_settings | الاستوديو/الثيمات/البانر |
| `MANAGE_STORIES` | manage_menu | الاستوري والبروفايل والأخبار |
| `USE_ASSISTANT` | (جديدة) | فتح المساعد الذكي وتنفيذ أوامره |
| `VIEW_REVENUE` | view_reports | رؤية أرقام المال (الرئيسية/التقارير تُخفي المبالغ بدونها) |
| `EDIT_PRICES` | manage_menu | تعديل الأسعار (يستطيع تحرير الاسم/الصورة بدون السعر) |
| `MANAGE_INVENTORY` | (جديدة) | المخزون والموردون وأوامر الشراء |
| `EXPORT_DATA` | view_reports | تصدير Excel/PDF للبيانات |
| `MANAGE_INTEGRATIONS` | manage_settings | مفاتيح الدفع/واتساب/البريد (أخطر جزء — يُعزل) |

الإعدادات تصبح مجموعة صلاحيات بدل صلاحية واحدة عملاقة، فيمكن منح «التسويق» دون «مفاتيح الدفع».

#### ب) إصلاح طبقة الإنفاذ (Firestore rules) — الأهم

مصدر الحقيقة للصلاحيات الفعّالة يُكتب في `staff/{uid}.caps` (المدير يكتبه أصلاً، والموظف ممنوع من تعديله — سطر 422/425). ثم في القواعد:

```
function staffCaps(tid) {
  return get(/databases/$(database)/documents/tenants/$(tid)/staff/$(request.auth.uid)).data.caps;
}
function hasCap(tid, cap) {
  return isManager(tid) || (isMember(tid) && cap in staffCaps(tid));
}
```

ثم استبدال `isManager(tid)` في الكتابات القابلة للتفويض بـ `hasCap(tid, 'manage_menu')` إلخ.
(يبقى `isManager` وحده للأشياء الأخطر: حذف الطلبات، الرواتب، حذف المنشأة، مفاتيح التكامل.)

**آلية المزامنة:** عند حفظ الأدوار أو تعديل موظف، نحسب الصلاحيات الفعّالة
(`capsFor(role, override, roleCaps)`) ونكتب المصفوفة الناتجة في `staff/{uid}.caps`.
و`auth.jsx` يقرأ الصلاحيات من وثيقة `staff/{uid}` (بدل `users` المحجوبة) — تعديل صغير في `loadContext`.

#### ج) محرّر صلاحيات فردي في `StaffProfile.jsx`

تبويب جديد «الصلاحيات» (للمدير فقط): قائمة كل الصلاحيات مع مفاتيح، تبدأ من الافتراضي حسب الدور،
والمدير يضيف/يزيل لهذا الشخص تحديداً → يُحفظ في `staff/{uid}.caps` (يعيد حساب المصفوفة الفعّالة).
هكذا: «موظف كاشير + أعطيه أيضاً manage_campaigns» أو «مشرف بلا view_revenue».

#### د) دور جاهز «تسويق» + بوابة مخصّصة

- إضافة `marketing` إلى `ASSIGNABLE_ROLES` وصلاحياته الافتراضية:
  `[MANAGE_CAMPAIGNS, VIEW_CUSTOMERS, MANAGE_LOYALTY, MANAGE_STORIES, MANAGE_OFFERS, USE_ASSISTANT, ATTENDANCE]`.
- التنقّل (`AdminLayout`) يظهر فقط الأقسام المسموحة (مبني أصلاً على `allowed()` — يحتاج ربط الأقسام الجديدة بالصلاحيات الجديدة).
- صفحة افتتاحية حسب الدور: التسويق يهبط على «الحملات»، الكاشير على «الكاشير»، لا على Dashboard المالي.

#### هـ) تقييد داخل القسم الواحد (sub-feature gating)

- **الرئيسية/التقارير:** إخفاء كل مبلغ مالي خلف `VIEW_REVENUE` (استبدال المبالغ بـ «—»).
- **الأصناف:** حقل السعر معطّل بدون `EDIT_PRICES`؛ زر الحذف بدون `MANAGE_MENU`.
- **الإعدادات:** كل تبويب يُحرَس بصلاحيته (المظهر=`MANAGE_APPEARANCE`، الولاء=`MANAGE_LOYALTY`، التكاملات=`MANAGE_INTEGRATIONS`).
- **العملاء:** التصدير خلف `EXPORT_DATA`؛ الأرقام/الإنفاق خلف `VIEW_REVENUE`.

#### و) خريطة القسم ← الصلاحية (المرجع الكامل)

| القسم | المسار | الحارس الحالي | المقترح |
|---|---|---|---|

### 1.4 مراحل التنفيذ المقترحة (بالترتيب، بلا كسر)

1. توسيع `permissions.js` (صلاحيات + دور marketing + تسميات) — لا يكسر شيئاً (إضافات).
2. مزامنة `staff/{uid}.caps` عند حفظ الأدوار/الموظف + قراءتها في `auth.jsx`.
3. محرّر الصلاحيات الفردي في `StaffProfile.jsx`.
4. تحديث `firestore.rules`: `hasCap()` + تفويض الكتابات القابلة للتفويض. **(ينشر المالك)**
5. ربط الأقسام الجديدة بالصلاحيات في `AdminLayout` + صفحة افتتاحية حسب الدور.
6. التقييد داخل الأقسام (السعر/المبالغ/تبويبات الإعدادات).
7. اختبار: موظف تسويق يرى الحملات فقط ويحفظ فعلاً؛ كاشير لا يرى الأرباح.

## القسم الثاني — حالة تحديث المساعد الذكي

المساعد حُدّث بالكامل في الدفعة السابقة (140 أداة، دليل 17 موضوعاً، سياق حي، ذاكرة دائمة).
**ملاحظة صدق:** الفحص المتعدد الوكلاء لكل أداة اصطدم بحدّ الجلسة؛ ما يلي مبني على قراءة البنية لا سطراً بسطر.
نقاط يجب التأكد من ربطها بأدوات المساعد (قد تكون ناقصة): أرشفة الأصناف، الإجراءات الجماعية،
دمج العملاء، تقرير المالك، حزم الاستوديو الجاهزة، بطاقة الجاهزية — أي ميزة أُضيفت حديثاً تحتاج أداة مقابلة.

### 20 اقتراحاً لتطوير المساعد (استخدامات جديدة داخل النظام)

**تشغيلية:**
1. **مساعد الكاشير الصوتي/النصي داخل شاشة الكاشير:** «أضف كابتشينو لطاولة 5» ينفّذ الطلب مباشرة.
2. **مناوب ذكي للطلبات المتأخرة:** ينبّه ويقترح إعادة ترتيب المطبخ حين يتكدّس KDS.
3. **مدقّق إغلاق اليومية:** يقارن درج النقد بالمبيعات ويكشف الفروقات ويكتب ملخّص إغلاق.

**تحليلي/استراتيجي:**
4. **تنبؤ الطلب:** يتوقّع مبيعات الغد/الأسبوع من التاريخ ويقترح كميات التحضير.
5. **محلّل التسعير:** يقترح أسعاراً حسب الهامش والمنافسين (بحث السوق) والمرونة.
6. **كاشف الهدر:** يربط المخزون بالمبيعات ويحدّد المواد قرب الانتهاء أو زائدة الشراء.
7. **تشريح السلّة:** «من اشترى X اشترى Y» ويقترح باقات/إقترانات تلقائية.
8. **رادار الانكماش:** يرصد العملاء المتناقص إنفاقهم ويطلق حملة win-back مستهدفة.

**تسويقي (يربط بموظف التسويق الجديد):**
9. **مؤلّف الحملات الكامل:** من فكرة → نص + جمهور + توقيت أمثل + A/B، جاهز للإرسال.
10. **مولّد محتوى الاستوري/المنشورات** بنبرة المنشأة وبيانات أصنافها الحقيقية.
11. **مخطّط تقويم تسويقي شهري** (مناسبات سعودية، رمضان، أعياد) بحملات مجدولة.

**دعم وتعليم (يدمج مع التيتوريال):**
12. **مساعد «كيف أفعل؟» سياقي:** يعرف الصفحة التي أنت فيها ويشرح خطواتها.
13. **مرشد الإعداد التفاعلي:** يكمل نقاط بطاقة الجاهزية بالحوار («أضف شعارك الآن»).
14. **مترجم الأخطاء:** يحوّل أي خطأ تقني إلى سبب وحل بالعربية.

**إدارة ذاتية للنظام:**
15. **مهندس الصلاحيات بالمحادثة:** «اعطِ سارة صلاحية الحملات فقط» ينفّذ فوراً (يربط بالقسم الأول).
16. **مصمّم القوائم بالذكاء:** استيراد منيو من صورة/PDF + تنظيم تلقائي بالتصنيفات والصور.
17. **مراقب الجودة:** فحص أسبوعي للأصناف بلا صور/وصف/سعر شاذ ويقترح إصلاحات.

**عميل/ولاء:**
18. **مساعد داخل المنيو للعميل** (اختياري للمنشأة): يوصي بأصناف حسب الذوق ويجيب أسئلة المكوّنات/الحساسية.
19. **مدير الولاء الذكي:** يقترح متى يُرقّى عميل ومتى تُرسل مكافأة، ويشغّلها تلقائياً.
20. **الملخّص الصباحي التنفيذي:** رسالة واحدة للمالك تجمع (أمس + تنبيهات + 3 قرارات مقترحة اليوم).

كل اقتراح يحترم العزل بين المنشآت والذاكرة الدائمة والبيانات الحقيقية (لا هلوسة)، ويُنفَّذ كأداة في `actions.js`.

## القسم الثالث — نظام التيتوريال والإرشادات (المؤجَّل سابقاً)

**الهدف:** كل صفحة تشرح نفسها، بلا قراءة دليل خارجي.

1. **جولة أول مرة (Product Tour):** طبقة تمييز متسلسلة عند أول دخول لكل صفحة رئيسية
   (تظليل العنصر + بطاقة شرح + «التالي»)، تُحفظ «شوهدت» في `localStorage` + `tenant.toursSeen`.
2. **زر «؟» في كل رأس صفحة:** يفتح شرحاً مختصراً (يعتمد `HelpTip.jsx` + `aiGuide.js` الموجودَين).
3. **حالات فارغة تعليمية:** كل قسم بلا بيانات يعرض «ابدأ بـ…» مع زر مباشر (بعضها موجود، نوحّدها).
4. **مركز مساعدة `/admin/help`:** فهرس مواضيع `SYSTEM_GUIDE` + بحث + فيديوهات قصيرة لاحقاً.
5. **ربط بالمساعد:** أي شرح فيه زر «اسأل المساعد» يفتح المحادثة بسياق الصفحة (اقتراح 12).
6. **شارة تقدّم التعلّم:** «أكملت 6 من 9 خطوات» تربط ببطاقة الجاهزية.

## القسم الرابع — النواقص لاكتمال النظام (شوائب يجب إغلاقها)

**حرجة/عالية:**
- إنفاذ الصلاحيات الدقيقة في القواعد (القسم الأول) — أهم فجوة.
- تعبئة `functions/.env` الفارغة (واتساب/بريد) وإلا الميزات المعتمدة عليها صامتة.
- التأكد من فهارس Firestore المركّبة للاستعلامات الجديدة (تقرير المالك يستخدم نطاق حقل واحد — سليم؛ يجب تدقيق باقي الاستعلامات).

**متوسطة:**
- لا اختبارات آلية إطلاقاً ولا مراقبة أخطاء إنتاجية موحّدة (يوجد `monitor.js`/`platformErrors` — يُوسّع).
- صفحة 404 مخصّصة + معالجة أخطاء الشبكة الموحّدة في الشاشات ذات onSnapshot.
- تقسيم الحزم (chunks > 500KB: firebase/charts/xlsx/pdf) بتحميل كسول أدق.
- سياسة نسخ احتياطي/تصدير بيانات المنشأة.

**منخفضة/تحسينية:**
- التيتوريال (القسم الثالث).
- توحيد الحالات الفارغة والتحميلية عبر كل الصفحات.
- مراجعة R1-R7 الأمنية المتبقية (تحقق `createPayIntent`، اشتقاق الإجماليات خادمياً).

## المخرجات التالية (تنتظر اختيارك)

الأولوية المقترحة: **1) نظام الصلاحيات الدقيق** (طلبك الأساسي) →
**2) التيتوريال** → **3) أدوات المساعد الجديدة**. ننفّذ واحدة تلو الأخرى مع بناء أخضر بعد كلٍّ.
